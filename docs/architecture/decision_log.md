# Архитектурный журнал решений (ADR)

Этот документ содержит запись архитектурных решений, принятых в проекте D&D MUD.

---

## ADR-001: Выбор Clean Architecture

**Статус:** Принято  
**Дата:** 2024-01-01  
**Автор:** Tomas

### Контекст
Проект представляет собой текстовую RPG игру, которая должна:
- Быть легко тестируемой
- Поддерживать различные UI (консоль, будущий web)
- Иметь чёткое разделение бизнес-логики и инфраструктуры
- Быть масштабируемой для будущих MUD-функций

### Решение
Применить Clean Architecture Роберта Мартина с 4 слоями:
1. **Entities** - бизнес-сущности и правила
2. **Use Cases** - сценарии использования  
3. **Interface Adapters** - преобразование данных
4. **Frameworks** - внешние зависимости

### Последствия
**Позитивные:**
- Высокая тестируемость каждого слоя
- Лёгкость замены UI фреймворков
- Чёткое разделение ответственности
- Поддержка Dependency Injection

**Негативные:**
- Больше файлов и директорий
- Необходимость в DTO для передачи данных
- Сложнее для новичков в архитектуре

## ADR-002: Использование Value Objects вместо простых типов

**Статус:** Принято  
**Дата:** 2024-01-01  
**Автор:** Tomas

### Контекст
В проекте есть сущности, которые должны быть валидируемы и иметь бизнес-правила:
- Язык (код и название)
- Контент приветствия
- ASCII-арт объекты

### Решение
Использовать Value Objects Pattern с базовым классом `BaseValidatable`:
```python
class BaseValidatable(ABC):
    @abstractmethod
    def validate(self) -> None: ...
```

### Последствия
**Позитивные:**
- Валидация инкапсулирована в объектах
- Повторное использование логики валидации
- Явные бизнес-правила
- Type safety

**Негативные:**
- Больше кода для простых сущностей
- Необходимость в dataclass

## ADR-003: Файловое хранилище вместо базы данных

**Статус:** Принято  
**Дата:** 2024-01-01  
**Автор:** Tomas

### Контекст
Для однопользовательской игры требуется хранение:
- Игровых данных (расы, классы, способности)
- Переводов интерфейса
- Сохранений игроков

### Решение
Использовать файловую систему:
- **YAML** для игровых данных (человеко-читаемый)
- **JSON** для переводов и сохранений
- **Python files** для констант

### Последствия
**Позитивные:**
- Простота разработки и отладки
- Версионный контроль данных
- Легкость модификации вручную
- Нет необходимости в сервере БД

**Негативные:**
- Медленнее при больших объёмах данных
- Проблемы с конкурентным доступом
- Ручное управление транзакциями

## ADR-004: Консольный UI как первый интерфейс

**Статус:** Принято  
**Дата:** 2024-01-01  
**Автор:** Tomas

### Контекст
Проект должен иметь минимально жизнеспособный продукт (MVP) с:
- Простым интерфейсом
- Фокусом на игровой логике
- Возможностью быстрой разработки

### Решение
Начать с консольного интерфейса используя:
- **Colorama** для цветного вывода
- **ASCII Art** для графики
- **Simple menus** для навигации

### Последствия
**Позитивные:**
- Быстрая разработка
- Фокус на логике игры
- Работает везде
- Низкие требования

**Негативные:**
- Ограниченный UX
- Меньше привлекательности
- Потребуется рефакторинг для web UI

## ADR-005: Строгая типизация и MyPy

**Статус:** Принято  
**Дата:** 2024-01-01  
**Автор:** Tomas

### Контекст
Проект должен быть надёжным и поддерживаемым:
- Предотвращение runtime ошибок
- Лучшая документация через типы
- Рефакторинг с уверенностью

### Решение
Использовать строгую типизацию:
- **Type hints** для всех функций
- **MyPy** для проверки типов
- **Dataclasses** для моделей данных
- **Generic types** где необходимо

### Последствия
**Позитивные:**
- Меньше runtime ошибок
- Лучшая IDE поддержка
- Самодокументирующийся код
- Уверенный рефакторинг

**Негативные:**
- Больше шаблонного кода
- Сложнее для начинающих
- Время на MyPy проверку

## ADR-006: pytest с маркерами для тестирования

**Статус:** Принято  
**Дата:** 2024-01-01  
**Автор:** Tomas

### Контекст
Проект требует надёжного тестирования:
- Разные типы тестов (unit, integration)
- Изоляция тестов
- Покрытие кода
- Легкость запуска конкретных тестов

### Решение
Использовать pytest с:
- **Маркерами** для группировки тестов
- **Fixtures** для setup/teardown
- **Coverage** для измерения покрытия
- **Mocking** для изоляции зависимостей

### Последствия
**Позитивные:**
- Чёткая организация тестов
- Легкий запуск конкретных типов
- Измерение покрытия
- Изоляция зависимостей

**Негативные:**
- Необходимо изучать pytest
- Больше конфигурации
- Возможные медленные тесты
