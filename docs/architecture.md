# Архитектура проекта

## Обзор архитектуры

D&D Text MUD построен на принципах чистой архитектуры с максимальным упрощением согласно KISS, YAGNI и Zen Python.

## Структура директорий

```
src/
├── entities/           # Бизнес-сущности D&D
│   └── character.py   # Персонаж с бизнес-логикой
├── use_cases/         # Сценарии использования
│   └── game.py        # Основная логика игры
├── interfaces/        # Интерфейсы
│   └── user_interface.py # UI контракт
├── ui/               # Пользовательский интерфейс
│   └── console.py    # Консольный интерфейс
├── core/             # Ядро системы
│   ├── game.py       # Координатор игры
│   └── constants.py  # Общие константы (DRY)
└── main.py          # Точка входа
```

## Основные компоненты

### 1. Entities - Бизнес-сущности

**Character** - основная бизнес-сущность персонажа:
- Характеристики: имя, уровень, HP
- Бизнес-правила: проверка жизни, статус персонажа
- Никаких внешних зависимостей
- Максимально простая реализация (KISS)

### 2. Use Cases - Сценарии использования

**GameUseCase** - основной сценарий игры:
- Показ и обработка главного меню
- Создание нового персонажа
- Простая логика без излишеств (YAGNI)
- Использует общие константы (DRY)

### 3. Interfaces - Интерфейсы

**UserInterface** - контракт UI:
- Абстрактные методы для вывода и ввода
- Типизированные сигнатуры
- Независимость от реализации

### 4. Infrastructure - Внешний мир

**Console** - консольный интерфейс:
- Простая реализация UserInterface
- Упрощённая поддержка цветов
- Минимальная функциональность

**Game** - координатор:
- Простой игровой цикл
- Обработка исключений
- Координация компонентов

**Constants** - общие константы:
- Все строковые константы в одном месте
- Следование DRY принципу
- Простые и понятные названия

## Поток данных

### 1. Запуск игры
```
main.py → Game.__init__() → GameUseCase.__init__() → Console.__init__()
```

### 2. Игровой цикл
```
Game.run() → GameUseCase.show_and_handle_menu() → Console методы
```

### 3. Создание персонажа
```
GameUseCase → Character(name) → валидация → отображение результата
```

## Принципы проектирования

### 1. Чистая архитектура (упрощённая)
- Entities не зависят от внешнего мира
- Use Cases зависят только от Entities
- UI зависит от Use Cases
- Минимум абстракций

### 2. KISS и YAGNI
- Простые решения вместо сложных
- Только необходимая функциональность
- Никакого кода "на будущее"
- Читаемость важнее абстракций

### 3. Zen Python
- Явное лучше неявного
- Простое лучше сложного
- Читаемость важна
- Плоское лучше вложенного

### 4. DRY принцип
- Общие константы в отдельном модуле
- Устранение дублирования кода
- Переиспользуемые компоненты

### 5. Type Safety
- Type hints везде
- MyPy для проверки типов
- Явные типы в сигнатурах

## Тестирование

### 1. Unit тесты
- Тестирование бизнес-логики Character
- Тестирование GameUseCase
- Тестирование Console
- Тестирование Game
- Мокирование зависимостей

### 2. Покрытие
- Все бизнес-методы протестированы
- UI методы протестированы
- Удалены избыточные тесты (YAGNI)

## Качество кода

### Инструменты
- **ruff**: линтинг и форматирование
- **mypy**: проверка типов
- **pytest**: тестирование
- **black**: форматирование кода

### Метрики
- Строки ≤ 79 символов
- Покрытие тестами > 80%
- Типизация всех методов
- Простые методы (<20 строк)

## Расширяемость

### Простое добавление
- Новые бизнес-сущности в entities/
- Новые сценарии в use_cases/
- Альтернативные UI в ui/

### Изменения
- Легко менять бизнес-правила
- Легко менять UI
- Легко добавлять новые сценарии

## Результаты рефакторинга

- **✅ Character**: Упрощена логика статуса
- **✅ Console**: Улучшена работа с цветами через константы
- **✅ Constants**: Создан общий модуль констант (DRY)
- **✅ GameUseCase**: Упрощена логика меню
- **✅ Тесты**: Удалены избыточные фикстуры и дублирующиеся тесты
- **✅ Форматирование**: Все строки ≤ 79 символов

## Философия проекта

**Максимальное упрощение** - лучше иметь простой работающий код, чем сложный потенциально расширяемый.

**Постепенное развитие** - добавлять сложность только когда она действительно нужна.

**Читаемость** - код должен быть понятным без комментариев.

**Простота** - следование Zen Python во всех решениях.
