# Product Requirements Document (PRD)
## Текстовая MUD-игра по мотивам Dungeons & Dragons 5-й редакции

**Версия документа:** 1.2
**Стадия:** Pre-Alpha
**Платформы:** Windows, Linux
**Инструменты:** Python 3.12, colorama, PyYAML, pyproject.toml
**БД:** YAML-справочники в `database/` (17 файлов) + JSON для настроек и сейвов

---

## 1. Цель проекта
Создать модульную, расширяемую текстовую приключенческую игру в жанре MUD, основанную на правилах D&D 5e, с фокусом на одиночное прохождение через выбор вариантов из нумерованных списков. Игра должна легко модифицироваться пользователями без навыков программирования через YAML-файлы.

## 2. Объем (Pre-Alpha — текущее состояние)

### Реализовано
- Приветственный экран (ASCII-art + colorama)
- Главное меню (5 пунктов + Выход: Новая игра, Загрузить игру, Создать персонажа, Настройки, Languages)
- Экран «Настройки» (отображение сложности по умолчанию; язык через Languages)
- Экран «Languages» (переключение языка интерфейса)
- Flow «Новая игра» (сложность → персонаж → приключение; запуск без полного engine)
- Flow «Создать персонажа» (сложность → имя → раса → подраса → генерация характеристик → класс → сохранение)
- База данных правил D&D 5e (YAML-справочники в `database/`)
- Локализация (русский/английский, YAML-словари)
- Броски кубиков (d20, ndm, преимущество/помеха, критические попадания)
- Модель персонажа (`core/models.py`: Character dataclass; сохранение через `core/character.py` в JSON)
- Загрузка приключений из YAML
- Модуль модов (сканирование, включение/выключение)
- Игровой движок (базовая структура)
- Адаптивный вывод текста с переносом по ширине терминала

### Запланировано (следующие итерации)
- Flow «Загрузить игру»: загрузка сохранённого состояния (сейчас — заглушка)
- Полноценный движок приключений (комнаты, проверки, бой)
- Фильтрация приключений и модов по режиму HardCore
- Режим сложности «Лёгкая» (`easy`)
- Автосохранение приключения
- Отдельное меню модификаций (опционально)

## 3. Функциональные требования

### 3.1. Приветственный экран
- Выводится при запуске игры.
- Текст: «Добро пожаловать в текстовую игру по мотивам Dungeons and Dragons 5 edition»
- Ниже — версия игры (из `pyproject.toml` или отдельного конфига).
- Использовать цветные ASCII-границы через `colorama`.

### 3.2. Главное меню (целевое)
Нумерованный список пунктов. По умолчанию от пользователя ожидается ввод **integer** из диапазона 0-5. Пункт "0" — выход из игры/приложения.

1. Новая игра
2. Загрузить игру
3. Создать персонажа
4. Настройки
5. Languages
0. Выход

**Текущее состояние:** Пункты 1, 3–5 реализованы. Пункт 2 («Загрузить игру») — заглушка.

#### 3.2.1. Режимы сложности игры

**Терминология**

- **Режим сложности игры** (`game difficulty`) — выбор игрока при «Новая игра» / «Создать персонажа». Хранится в `Character.difficulty` и `settings.json` → `difficulty`.
- **Уровень контента приключения** — поле `difficulty` в `database/content/adventures.yaml` (`easy`, `normal`). Описательная метка контента в каталоге; **не** режим игрока. Целевое имя поля при рефакторинге YAML: `content_tier`.

**Текущие режимы (Pre-Alpha)**

| ID | UI (ru) | Описание |
|----|---------|----------|
| `normal` | Нормальная | Полный контроль над характеристиками (3 метода), переквалификация; базовая механика |
| `hardcore` | HardCore | Авто-4d6×6, без переквалификации; **флаг «полной» игры** |

**Планируемый режим**

- `easy` (Лёгкая) — зарезервирован в спецификации; не реализован в UI; поведение TBD (упрощённая механика / обучение).

**HardCore — ключевой режим (целевая модель)**

Режим HardCore — центральная ось проекта: от него завязаны игровая механика, доступность приключений и модов.

- Хранится в `Character.difficulty` и наследуется на всю сессию.
- **Создание персонажа**: см. §3.4.5.
- **Приключения** (запланировано): каталог поддерживает список допустимых режимов, напр. `allowed_game_difficulties: [normal, hardcore]` или `hardcore_only: true`; UI фильтрует приключения по режиму персонажа/сессии. Сейчас показываются все приключения без фильтрации.
- **Моды** (запланировано): в метаданных мода — требование режима, напр. `requires_game_difficulty: hardcore`; моды HardCore-only не активны в Normal.
- **Game engine** (запланировано): проверки DC, урон, смерть, ограничения ресурсов параметризуются режимом; HardCore = полные правила D&D 5e без упрощений (Phase 2).

**Где выбирается**

- Flow «Новая игра» §3.3.1 и «Создать персонажа» §3.4.1 — один экран `select_difficulty()` (`ui/menus.py`).
- `settings.json` → `difficulty` — значение по умолчанию (предвыбор `*` в меню).
- Меню «Настройки» §3.5 только **отображает** сложность по умолчанию, не переключает её.

### 3.3. Flow «Новая игра»
При выборе пункта «Новая игра» пользователь последовательно проходит через экраны:

#### 3.3.1. Выбор сложности
- См. §3.2.1 «Режимы сложности игры».
- Отображается список режимов сложности:
  1. Нормальная (`normal`)
  2. HardCore (`hardcore`)
  0. Назад
- В будущем может быть добавлен пункт «Лёгкая» (`easy`).
- Тип ввода: **integer** по нумерованному списку.
- Сложность передаётся дальше по flow и сохраняется в `Character.difficulty`.
- Если в «Настройках» задана сложность по умолчанию, она используется как предвыбор (`*`).

#### 3.3.2. Выбор персонажа
- Отображается список сохранённых персонажей с полями: Имя, Уровень, Раса, Класс.
- Тип ввода: **integer** по нумерованному списку.
- Если нет сохранённых персонажей — предложить создать нового (переход на flow «Создать персонажа»).
- Пункт "0" — Назад (к выбору сложности) **всегда последний**.
- Выбранный персонаж должен соответствовать минимальному уровню выбранного приключения.

#### 3.3.3. Выбор приключения
- Список приключений (из `database/content/adventures.yaml`) с отображением: Название, тип (обучение/сюжетное/короткое), уровень (от-до), уровень контента (`difficulty` в YAML — см. §3.2.1).
- **Запланировано**: фильтрация по режиму игры (`allowed_game_difficulties` / `hardcore_only` в YAML); сейчас показываются все приключения.
- Тип ввода: **integer** по нумерованному списку.
- Уровень персонажа должен точно соответствовать минимальному уровню приключения.
- Пункт "0" — Назад (к выбору персонажа) **всегда последний**.
- При выборе запускается игровой цикл.

### 3.4. Flow «Создать персонажа»
При выборе пункта «Создать персонажа» пользователь последовательно проходит через экраны:

**Порядок шагов:** сложность → имя → раса → подраса → генерация характеристик → класс → сохранение.

#### 3.4.1. Выбор сложности
- См. §3.2.1 «Режимы сложности игры».
- Отображается список режимов сложности:
  1. Нормальная (`normal`)
  2. HardCore (`hardcore`)
  0. Назад
- В будущем может быть добавлен пункт «Лёгкая» (`easy`).
- Тип ввода: **integer** по нумерованному списку.
- Сложность сохраняется в `Character.difficulty`.

#### 3.4.2. Ввод имени персонажа
- Запрашивается имя персонажа (тип ввода: **string**).
- Имя должно состоять из одного слова.
- Пробелы автоматически удаляются.
- Разрешены только буквы кириллицы и латиницы.
- Запрещены специальные символы и цифры.
- Длина не менее 2 символов.
- При неверном формате выводится ошибка с пояснением, повторный запрос.

#### 3.4.3. Выбор расы
- Загружаются данные из `database/races/races.yaml`.
- Модель Race создаётся конструктором на основе YAML-данных с использованием `dataclasses`.
- Отображается нумерованный список доступных рас.
- Каждая раса в списке содержит:
  - Номер и имя расы
  - Краткое описание (description)
  - Бонусы к характеристикам (ability_bonuses)
  - Особенности (features) — список названий особенностей
- Пример отображения:
```
  ============================================
                   ВЫБОР РАСЫ

1. Человек
   Человечество — самая разнообразная и амбициозная раса
   Бонусы: Сила+1, Ловкость+1, Телосложение+1, Интеллект+1, Мудрость+1, Харизма+1
   Особенности:
     • Дополнительный язык

2. Эльф
   Эльфы — изящные и долгоживущие существа с тонкими чувствами
   Бонусы: Ловкость+2
   Особенности:
     • Обострённые чувства (Внимательность)
     • Тёмное зрение (60 футов)
     • Наследие фей (преимущество против очарования)
     • Транс (4 часа медитации)

0. Назад
```
- Тип ввода: **integer** по нумерованному списку.

#### 3.4.4. Выбор подрасы
- Если у расы есть подрасы — экран описания и выбор подрасы (включая variant human).
- Если подрас нет — переход к генерации характеристик.

#### 3.4.5. Генерация характеристик (реализовано)

Шаг выполняется после выбора расы и подрасы; `race_id` и `subrace_id` уже известны.

**Порядок характеристик** (фиксированный, `STAT_NAMES` в `core/character.py`):

Сила → Ловкость → Телосложение → Интеллект → Мудрость → Харизма

> В UI локализация CON отображается как «Выносливость» (`database/strings/ru.yaml`); в PRD используется канон D&D 5e RU — «Телосложение».

**Общие правила** (для всех режимов — см. §3.2.1)

- **Перед распределением / бросками**: показать суммарные бонусы от расы и подрасы через `get_race_bonuses()` (например, «Расовые бонусы: Сила+1, …»).
- **После распределения**: экран итоговых значений **с уже применёнными** расовыми бонусами (база + бонус; подпись «Итого (с расовыми бонусами)»).
- Функции `generate_stats_*` в `core/character.py` возвращают финальные значения (база + расовый бонус).

**Режим Normal** (`normal`) — см. §3.2.1

После показа расовых бонусов — меню выбора метода (integer, пункт `0` — Назад):

1. **Стандартный массив** `[15, 14, 13, 12, 10, 8]`
   - Пользователь по очереди **вводит** значение для каждой характеристики (не выбор из нумерованного списка).
   - На экране отображается обновляемый пул «Доступные значения»; ввод `0` — назад.
   - Каждое число из пула используется ровно один раз.

2. **Покупка очков (Point-buy)** — бюджет **27 очков**
   - Стартовые значения: 8 для всех шести характеристик.
   - Пользователь выбирает характеристику (1–6) и **вводит** новое значение (8–15); можно как повышать, так и понижать.
   - На экране отображается «Доступно очков: X / 27» после каждого изменения.

   | Значение | Стоимость |
   |----------|-----------|
   | 8 | 0 |
   | 9 | 1 |
   | 10 | 2 |
   | 11 | 3 |
   | 12 | 4 |
   | 13 | 5 |
   | 14 | 7 |
   | 15 | 9 |

3. **Случайный (4d6)**
   - Алгоритм `roll_ability_score()` (`core/dice.py`): бросить 4d6 → убрать наименьший результат → сложить остальные три.
   - Повторить **6 раз** → пул из 6 значений.
   - Экран бросков: показать массив результатов; меню `1` — **Принять броски**, `2` — **Перегенерировать**, `0` — назад.
   - После принятия — распределение пула по характеристикам через нумерованный список (с учётом дубликатов бросков).

**Экран подтверждения** (только Normal, после любого из трёх методов):

- Показать итоговые значения с расовыми бонусами.
- Опции: `1` — принять; `2` — **Переквалификация** (вернуться к выбору метода, перераспределить или перегенерировать); `0` — назад.

**Режим HardCore** (`hardcore`) — см. §3.2.1

- Меню методов **не показывается**.
- **Перед бросками**: показать расовые бонусы (как в Normal).
- Для **каждой** характеристики по порядку (Сила … Харизма): один бросок 4d6 drop lowest → результат **сразу** присваивается текущей характеристике (6 бросков, без ручного распределения).
- Показать итоговые значения с расовыми бонусами.
- Экран подтверждения и переквалификация **недоступны**.

#### 3.4.6. Выбор класса
- Загружаются данные из `database/classes/classes.yaml`.
- Класс выбирается **после** финального принятия характеристик.
- Тип ввода: **integer** по нумерованному списку.

- После выбора класса персонаж сохраняется в `saves/characters.json` с указанной сложностью.

### 3.5. Меню «Настройки» (реализовано)
- Отображает текущую сложность по умолчанию из `settings.json` → `difficulty` (`normal` / `hardcore`); **не переключает** её.
- Выбор режима сложности — только в flow «Новая игра» / «Создать персонажа» (§3.3.1, §3.4.1).
- Язык интерфейса — в меню «Languages».

### 3.6. Меню «Languages» (реализовано)
- Самописная локализация на YAML-словарях (`load_strings`, `get_string`). Пакет `babel` в зависимостях не используется в коде.
- Язык по умолчанию: русский.
- Значки отображаются согласно текущей локализации:
  - При русской локализации:
    1. English
    2. Русский
    0. Назад
  - При английской локализации:
    1. Русский
    2. English
    0. Назад
- Тип ввода: **integer** по нумерованному списку.
- При смене языка интерфейс переключается с немедленным обновлением всех текстов.

### 3.7. Flow «Загрузить игру»
- Отображается список сохранённых приключений/сессий.
- Тип ввода: **integer** по нумерованному списку.
- Пункт "0" — Назад **всегда последний**.
- При выборе загружается состояние и запускается игровой цикл с момента сохранения.

## 4. Нефункциональные требования

### 4.1. Модульность
- Ядро (core) не зависит от UI, БД, модов.
- Core предоставляет API для:
  - Получения данных о персонаже
  - Выполнения действия (бросок кубика, проверка навыка, получение описания комнаты)
  - Управления состоянием игры (сохранение/загрузка)
- UI (меню, ввод/вывод) — отдельный слой.

### 4.2. Обработка ввода

**Основное правило:**  
По умолчанию от пользователя ожидается ввод **integer** соответствующего пункта нумерованного списка меню или списка действий. Первый пункт любого меню имеет номер "1", последний пункт "Выход" или "Назад" — номер "0". **Пункт "0" (Назад/Выход) должен отображаться всегда последним в любом меню.**

**Формат запроса:**  
При отрисовке меню используется строка вида:  
`Выберите пункт:`  

**Исключения (string ввод):**  
- Ввод имени персонажа при создании (не менее 2 символов, без пробелов и спецсимволов).
- Ввод пути к файлу мода (при необходимости).
- Ввод текста для поиска (в будущих итерациях).

**Валидация ввода:**  
- При integer вводе: проверка диапазона, повторный запрос при ошибке.
- При string вводе: проверка минимальной длины/формата, повторный запрос при ошибке.

### 4.3. Производительность
- Загрузка YAML-файлов при старте игры (или при смене языка/модов).
- Игровой цикл не должен задерживать ввод (отсутствие `time.sleep` без причины).

### 4.4. Локализация
- Все пользовательские тексты вынесены в YAML-словари (папка `database/strings/`).
- Ключи локализации используют точечную нотацию для вложенных словарей.
- При смене языка перезагружаются все текстовые ресурсы.
- Fallback на английский язык, если ключ не найден.

**database/content/adventures.yaml**

```yaml
adventures:
  - id: tutorial
    name:
      ru: "Обучение"
      en: "Tutorial"
    type: training
    difficulty: easy          # уровень контента (content tier), НЕ режим игрока — см. §3.2.1
    allowed_game_difficulties: [normal, hardcore]  # целевое поле: допустимые режимы игры
    script_file: "adventures_scripts/tutorial.yaml"
  - id: lost_mine
    name:
      ru: "Затеряные рудники Фанделвера"
      en: "Lost Mine of Phandelver"
    type: adventure
    difficulty: normal          # уровень контента
    allowed_game_difficulties: [hardcore]           # только HardCore (целевая модель)
    hardcore_only: true         # альтернативная запись (целевая модель)
    script_file: "adventures_scripts/lost_mine.yaml"
```

> Поля `allowed_game_difficulties` и `hardcore_only` — целевая схема; в runtime пока не загружаются. Текущий каталог использует только `difficulty` как content tier.

## 6. Архитектура (модули)

### 6.1. Core (ядро)
- `models.py` – dataclass `Character`, `Adventure`; `to_dict()` / `from_dict()`.
- `character.py` – `save_character()`, `load_characters()`, генерация характеристик, загрузка рас/классов из YAML.
- `dice.py` – `roll()`, `roll_d20()`, `roll_with_mods()`, `ability_modifier()`, `critical_hit()`.
- `adventure.py` – `load_adventures() -> list[Adventure]`; имя через `Adventure.get_name()`.
- `mod_loader.py` – сканирование `mods/`, включение/выключение модов.
- `localization.py` – `load_strings()`, `get_string()` (YAML-словари, fallback на английский).
- `settings.py` – `load_settings()`, `save_settings()`; JSON в `database/core/settings.json`.
- `game_engine.py` – `run_game_loop()`, `ability_check()` (заготовка, не вызывается из UI).

### 6.2. UI
- `menus.py` – функции отрисовки экранов (show_welcome_screen, show_main_menu, show_settings).
- `input_handler.py` – get_int_input(), get_str_input(), get_choice().

### 6.3. Entry point (main.py)
- Инициализация colorama, загрузка настроек, локализация.
- Запуск основного цикла меню (пока не выбран выход).

## 7. Пользовательский интерфейс

Все экраны центрированы или выровнены по левому краю с рамкой (ASCII-символы).

Каждый пункт меню предваряется номером.

**Правило порядка пунктов:**  
Пункт "0" (Назад/Выход) **всегда отображается последним** в любом меню или списке выбора.

Цвета: заголовки — жёлтый, ошибки — красный, успех — зелёный (через colorama).

Новый экран дописывается в конец вывода; терминал не очищается — пользователь может прокрутить историю и увидеть предыдущие шаги.

## 8. Приоритеты для Pre-Alpha

| Приоритет | Компонент | Статус |
|-----------|-----------|--------|
| P0 | Приветственный экран + главное меню | ✅ Реализовано |
| P0 | Flow «Новая игра» (сложность → персонаж → приключение) | ✅ UI реализован; engine — частично |
| P0 | Flow «Создать персонажа» (сложность → создание) | ✅ Реализовано |
| P0 | Flow «Загрузить игру» | ⏳ Заглушка |
| P1 | Настройки (отображение сложности по умолчанию, язык) | ✅ Реализовано |
| P1 | Меню Languages (русский/английский) | ✅ Реализовано |
| P2 | HardCore gating: фильтрация приключений и модов по режиму | ⏳ Запланировано |
| P2 | Полноценный движок приключений (комнаты, проверки, бой) | ❌ Не реализовано |
| P2 | Автосохранение/ручное сохранение приключения | ❌ Не реализовано |

## 9. Ограничения и допущения

- Одиночная игра (без сети).
- Нет поддержки многопользовательской игры.
- Движок боя — упрощённый (D20 + модификаторы).
- Состояние игры сохраняется только вручную (автосохранение не требуется на Pre-Alpha).
- Графика отсутствует (только текст).
- Моды не требуют перезагрузки интерпретатора (но могут требовать перезагрузки ядра через вызов функции reload_mods).
