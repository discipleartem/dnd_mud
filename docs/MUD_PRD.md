# Product Requirements Document (PRD)
## Текстовая MUD-игра по мотивам Dungeons & Dragons 5-й редакции

**Версия документа:** 1.5
**Стадия:** Pre-Alpha
**Платформы:** Windows, Linux
**Инструменты:** Python 3.12, colorama, PyYAML, pyproject.toml
**БД:** YAML-справочники в `database/` (17 файлов) + JSON для настроек и сейвов

---

## 1. Цель проекта
Создать модульную, расширяемую текстовую приключенческую игру в жанре MUD, основанную на правилах D&D 5e, с фокусом на одиночное прохождение через выбор вариантов из нумерованных списков. Игра должна легко модифицироваться пользователями без навыков программирования через YAML-файлы.

## 2. Объем (Pre-Alpha — текущее состояние)

### Реализовано
- Приветственный экран (ASCII-art + colorama)
- Главное меню (5 пунктов + Выход: Новая игра, Загрузить игру, Создать персонажа, Настройки, Languages)
- Экран «Настройки» (заглушка: только «Назад»; язык через Languages)
- Экран «Languages» (переключение языка интерфейса)
- Flow «Новая игра» (персонаж → приключение; режим из `Character.difficulty`; запуск без полного engine)
- Flow «Создать персонажа» (сложность → имя → раса → подраса → генерация характеристик → класс → подкласс (по режиму) → сохранение)
- Режим сложности «Лёгкая» (`easy`): старт с 3 уровня, обязательный выбор подкласса
- Выбор подкласса по режимам (`normal` / `hardcore` / `easy`); NPC-наставник в меню персонажей и сценариях
- Прогрессия XP и потолок уровня **10** (`MAX_CHARACTER_LEVEL`)
- Минимальный scenario runner (`core/scenario.py`): grant XP, subclass_training; tutorial / lost_mine
- База данных правил D&D 5e (YAML-справочники в `database/`)
- Локализация (русский/английский, YAML-словари)
- Броски кубиков (`roll`, `roll_ability_score`, `ability_modifier` в `core/dice.py`)
- Модель персонажа (`core/models.py`: Character dataclass; сохранение через `core/character.py` в JSON)
- Загрузка приключений из YAML
- Адаптивный вывод текста с переносом по ширине терминала

### Запланировано (следующие итерации)
- Flow «Загрузить игру»: загрузка сохранённого состояния (сейчас — заглушка)
- Полноценный движок приключений (комнаты, проверки, бой)
- Фильтрация приключений и модов по режиму HardCore (core/UI готовы; ограничения в YAML каталоге не заданы)
- Автосохранение приключения
- Отдельное меню модификаций (опционально)

Справочник правил — [`DND_RULES.md`](DND_RULES.md) и `docs/rules/`. Ниже перечислено то, что **описано в docs по PHB, но ещё не реализовано в игре** — это запланированный функционал, а не расхождение с каноном.

## 3. Функциональные требования

### 3.1. Приветственный экран
- Выводится при запуске игры.
- Текст: «Добро пожаловать в текстовую игру по мотивам Dungeons and Dragons 5 edition»
- Ниже — версия игры (из `pyproject.toml` или отдельного конфига).
- Использовать цветные ASCII-границы через `colorama`.

### 3.2. Главное меню (целевое)
Нумерованный список пунктов. По умолчанию от пользователя ожидается ввод **integer** из диапазона 0-5. Пункт "0" — выход из игры/приложения.

1. Новая игра
2. Загрузить игру
3. Создать персонажа
4. Настройки
5. Languages
0. Выход

**Текущее состояние:** Пункты 1, 3–5 реализованы. Пункт 2 («Загрузить игру») — заглушка.

#### 3.2.1. Режимы сложности игры

**Терминология**

- **Режим сложности игры** (`game difficulty`) — выбор игрока при «Новая игра» / «Создать персонажа». Хранится в `Character.difficulty`.
- **Уровень контента приключения** — поле `difficulty` в `database/content/adventures.yaml` (`easy`, `normal`). Описательная метка контента в каталоге; **не** режим игрока. Целевое имя поля при рефакторинге YAML: `content_tier`.

**Текущие режимы (Pre-Alpha)**

| ID | UI (ru) | Описание |
|----|---------|----------|
| `normal` | Нормальная | Полный контроль над характеристиками (3 метода), переквалификация; подкласс обязателен при создании |
| `hardcore` | HardCore | Авто-4d6×6, без переквалификации; подкласс при создании только если `subclass_choice_level ≤ 1` |
| `easy` | Лёгкая | Старт с **3 уровня**; подкласс обязателен при создании; упрощённый режим обучения |

**Потолок уровня:** в runtime реализованы только уровни **1–10** (`MAX_CHARACTER_LEVEL`). Умения и прогрессия с `level > 10` в YAML — справочные.

**Выбор подкласса по режимам**

| Режим | Стартовый уровень | Подкласс при создании | Активация |
|-------|-------------------|------------------------|-----------|
| `easy` | 3 | Обязателен | Сразу (`subclass_id` активен) |
| `normal` | 1 | Обязателен | При `level ≥ subclass_choice_level` |
| `hardcore` | 1 | Только если `subclass_choice_level ≤ 1` (жрец) | Иначе — NPC-наставник на 3+ ур. |

**HardCore — ключевой режим (целевая модель)**

Режим HardCore — центральная ось проекта: от него завязаны игровая механика, доступность приключений и модов.

- Хранится в `Character.difficulty` и наследуется на всю сессию.
- **Создание персонажа**: см. §3.4.6.
- **Приключения**: поля `allowed_game_difficulties` / `hardcore_only` / `min_level` загружаются в `Adventure` и проверяются в UI (`_select_adventure`). HardCore-персонаж не блокируется на приключениях без требования HardCore; недоступные — серым списком с причиной.
- **Моды** (запланировано): в метаданных мода — требование режима, напр. `requires_game_difficulty: hardcore`; моды HardCore-only не активны в Normal.
- **Game engine** (запланировано): проверки DC, урон, смерть, ограничения ресурсов параметризуются режимом; HardCore = полные правила D&D 5e без упрощений (Phase 2).

**Где выбирается**

- Flow «Создать персонажа» §3.4.1 — экран `select_difficulty()` (`ui/menus.py`).
- Flow «Новая игра» §3.3 — режим берётся из выбранного `Character.difficulty` (без отдельного экрана сложности).
- Меню «Настройки» §3.5 — заглушка (только «Назад»); язык — в «Languages» §3.6.

### 3.3. Flow «Новая игра»
При выборе пункта «Новая игра» пользователь последовательно проходит через экраны:

#### 3.3.1. Выбор персонажа
- Отображается список сохранённых персонажей с полями: Имя, Уровень, Раса, Класс.
- Тип ввода: **integer** по нумерованному списку.
- Если нет сохранённых персонажей — предложить создать нового (переход на flow «Создать персонажа»).
- Пункт "0" — Назад (к главному меню) **всегда последний**.
- Режим сложности сессии — `Character.difficulty` выбранного персонажа (задаётся при создании, §3.4.1).

#### 3.3.2. Выбор приключения
- Список приключений (из `database/content/adventures.yaml`) с отображением: Название, тип (обучение/сюжетное/короткое), уровень (от-до), уровень контента (`difficulty` в YAML — см. §3.2.1).
- **Реализовано**: фильтрация по `min_level` и режиму игры (`adventure_unavailable_reason`); недоступные приключения — серым списком с причиной.
- Тип ввода: **integer** по нумерованному списку.
- Уровень персонажа должен быть не ниже `min_level` приключения.
- Пункт "0" — Назад (к выбору персонажа) **всегда последний**.
- При выборе запускается игровой цикл.

### 3.4. Flow «Создать персонажа»
При выборе пункта «Создать персонажа» пользователь последовательно проходит через экраны:

**Порядок шагов:** сложность → имя → раса → подраса → генерация характеристик → **предыстория** → **языки** → класс → подкласс (если предусмотрен режимом) → **владения** → **навыки** → (компетентность?) → сохранение.

#### 3.4.1. Выбор сложности
- См. §3.2.1 «Режимы сложности игры».
- Отображается список режимов сложности:
  1. Лёгкая (`easy`)
  2. Нормальная (`normal`)
  3. HardCore (`hardcore`)
  0. Назад
- Тип ввода: **integer** по нумерованному списку.
- Сложность сохраняется в `Character.difficulty`.

#### 3.4.2. Ввод имени персонажа
- Запрашивается имя персонажа (тип ввода: **string**).
- Имя должно состоять из одного слова.
- Пробелы автоматически удаляются.
- Разрешены только буквы кириллицы и латиницы.
- Запрещены специальные символы и цифры.
- Длина не менее 2 символов.
- При неверном формате выводится ошибка с пояснением, повторный запрос.

#### 3.4.3. Выбор расы
- Загружаются данные из `database/races/races.yaml`.
- Отображается нумерованный список доступных рас.
- Каждая раса в списке содержит:
  - Номер и имя расы
  - Краткое описание (description)
  - Бонусы к характеристикам (ability_bonuses)
  - Особенности (features) — список названий особенностей
- Пример отображения:
```
  ============================================
                   ВЫБОР РАСЫ

1. Человек
   Человечество — самая разнообразная и амбициозная раса
   Бонусы: Сила+1, Ловкость+1, Телосложение+1, Интеллект+1, Мудрость+1, Харизма+1
   Особенности:
     • Дополнительный язык

2. Эльф
   Эльфы — изящные и долгоживущие существа с тонкими чувствами
   Бонусы: Ловкость+2
   Особенности:
     • Обострённые чувства (Внимательность)
     • Тёмное зрение (60 футов)
     • Наследие фей (преимущество против очарования)
     • Транс (4 часа медитации)

0. Назад
```
- Тип ввода: **integer** по нумерованному списку.

**Ограниченный каталог (Pre-Alpha).** В базовой поставке — подмножество PHB (см. [`02-races.md`](rules/02-races.md)): `human`, `elf`, `dwarf`, `half_orc` с подрасами из PHB 2023 RUS в YAML. Остальные расы PHB **осознанно не включены** в ядро; расширение — через механизм **Addon** (официальное дополнение контента, `type: addon` в метаданных мода, §5.4).

**Особенности рас (features).** На экране выбора показываются название и описание из YAML. **Владения** из `mechanics` (оружие, доспехи, инструменты) применяются через `core/proficiencies.py`. Прочая механика (тёмное зрение, сопротивления, владения **навыками** lore college и т.д.) — Phase 2. Исключение: выборные бонусы к характеристикам (`type: ability_bonus`, напр. variant human) — реализованы в flow генерации stats (§3.4.6).

#### 3.4.4. Выбор подрасы
- Если у расы есть подрасы — экран описания и выбор подрасы (включая variant human).
- Если подрас нет — переход к §3.4.6 (характеристики).

#### 3.4.6. Генерация характеристик (реализовано)

Шаг выполняется после выбора расы и подрасы; `race_id` и `subrace_id` уже известны.

**Порядок характеристик** (фиксированный, `STAT_NAMES` в `core/stats.py`):

Сила → Ловкость → Телосложение → Интеллект → Мудрость → Харизма

**Общие правила** (для всех режимов — см. §3.2.1)

- На **каждом** экране «ГЕНЕРАЦИЯ ХАРАКТЕРИСТИК» внутри выбранного метода (кроме меню выбора метода): показать суммарные бонусы от расы и подрасы через `get_race_bonuses()` (например, «Расовые бонусы: Сила+1, …»).
- **После распределения**: экран итоговых значений **с уже применёнными** расовыми бонусами (база + бонус; подпись «Итого (с расовыми бонусами)»).
- Функции `generate_stats_*` в `core/character.py` возвращают финальные значения (база + расовый бонус).
- **Выборные бонусы** (напр. variant human): после распределения базовых значений UI вызывает `get_choice_ability_bonus_mechanics()` и экран выбора характеристик; итог — через `apply_bonuses_to_stats()` / `get_effective_race_bonuses()`.

**Режим Normal** (`normal`) — см. §3.2.1

Меню выбора метода (integer, пункт `0` — Назад):

1. **Стандартный массив** `[15, 14, 13, 12, 10, 8]`
   - Пользователь по очереди **вводит** значение для каждой характеристики (не выбор из нумерованного списка).
   - На экране отображается обновляемый пул «Доступные значения»; ввод `0` — назад.
   - Каждое число из пула используется ровно один раз.

2. **Покупка очков (Point-buy)** — бюджет **27 очков**
   - Стартовые значения: 8 для всех шести характеристик.
   - Пользователь выбирает характеристику (1–6) и **вводит** новое значение (8–15); можно как повышать, так и понижать.
   - На экране отображается «Доступно очков: X / 27» после каждого изменения.

   | Значение | Стоимость |
   |----------|-----------|
   | 8 | 0 |
   | 9 | 1 |
   | 10 | 2 |
   | 11 | 3 |
   | 12 | 4 |
   | 13 | 5 |
   | 14 | 7 |
   | 15 | 9 |

3. **Случайный (4d6)**
   - Алгоритм `roll_ability_score()` (`core/dice.py`): бросить 4d6 → убрать наименьший результат → сложить остальные три.
   - Повторить **6 раз** → пул из 6 значений.
   - Экран бросков: показать массив результатов; меню `1` — **Принять броски**, `2` — **Перегенерировать**, `0` — назад.
   - После принятия — распределение пула по характеристикам через **ввод значения** (как стандартный массив): на экране пул «Доступные значения», ввод `0` — назад.

**Экран подтверждения** (только Normal, после любого из трёх методов):

- Показать итоговые значения с расовыми бонусами.
- Опции: `1` — принять; `2` — **Переквалификация** (вернуться к выбору метода, перераспределить или перегенерировать); `0` — назад.

**Режим HardCore** (`hardcore`) — см. §3.2.1

- Меню методов **не показывается**.
- **Перед бросками**: показать расовые бонусы (как в Normal).
- Для **каждой** характеристики по порядку (Сила … Харизма): один бросок 4d6 drop lowest → результат **сразу** присваивается текущей характеристике (6 бросков, без ручного распределения).
- Показать итоговые значения с расовыми бонусами.
- Экран подтверждения и переквалификация **недоступны**.

#### 3.4.7. Выбор предыстории (реализовано)
- Источник: `database/backgrounds/backgrounds.yaml` (13 PHB).
- Экран по образцу выбора расы: имя, описание, 2 навыка, умение (feature).
- После выбора: `background_id` → JSON `"background"`; навыки предыстории добавляются в state для шага навыков.
- Предыстория выбирается **до** класса (PHB: навыки предыстории учитываются при выборе классовых навыков).
- Инструменты предыстории — `tool_proficiencies` в YAML; выбор на шаге владений (§3.4.12). Стартовое снаряжение и механика feature — Phase 2.
- См. [`04-backgrounds.md`](rules/04-backgrounds.md).

#### 3.4.8. Языки (реализовано)
- Шаг **после предыстории**: фиксированные языки расы + выборные из `features` + языки предыстории (`languages.count > 0`).
- Пул по умолчанию — **common**; exotic — только при явном `pool` в YAML.
- Каталог: `database/core/languages.yaml`; логика: `core/languages.py`, UI: `ui/menus/languages.py`.
- Результат — `Character.languages` (JSON `"languages"`).

#### 3.4.9. Выбор класса
- Загружаются данные из `database/classes/classes.yaml`.
- Класс выбирается **после** финального принятия характеристик.
- Экран по образцу выбора **расы**: краткий обзор каждого класса (`description`, кость хитов, ключевая характеристика, владения, навыки, классовые умения до 10 ур.).
- Тип ввода: **integer** по нумерованному списку.
- После выбора класса: если `subclass_offered_at_creation(difficulty, class_id)` — переход к §3.4.10; иначе §3.4.12 (владения) → §3.4.11 (навыки).

**Один класс на персонажа.** Мультиклассирование **запрещено** на текущем этапе: при создании выбирается один класс (`class_name`), при повышении уровня класс не меняется. См. [`06-multiclass.md`](rules/06-multiclass.md).

**Ограниченный каталог (Pre-Alpha).** В базовой поставке — четыре класса PHB (см. [`03-classes.md`](rules/03-classes.md)): `fighter`, `rogue`, `cleric`, `bard` с подклассами **только из PHB 2023 RUS** (воин: мастер боевых искусств, мистический рыцарь, чемпион; плут: вор, убийца, мистический ловкач; жрец: домены жизни, обмана, света; бард: коллегии доблести и знаний). Остальные классы **осознанно не включены**; расширение — через **Addon** (§5.4).

**Особенности классов.** Поля `features`, `subclasses`, `skill_choices`, стартовое снаряжение в YAML — источник для UI. Применение умений в бою, ячейки заклинаний — **запланированы** (Phase 2). Выбор навыков — §3.4.11; подклассы — §3.4.10 и [`03-subclasses.md`](rules/03-subclasses.md).

#### 3.4.10. Выбор подкласса (реализовано)
- Экран по образцу выбора **подрасы**: полный обзор выбранного класса + список архетипов с деталями (`description`, умения подкласса до 10 ур.).
- Условия показа — см. таблицу режимов в §3.2.1.
- `subclass_id` сохраняется в JSON (`"subclass"`); на карточке персонажа — с пометкой «(с N ур.)», если ещё не активен.
- Повторный выбор: меню персонажей «Получить подкласс у наставника» (`subclass_trainer`) или действие сценария `subclass_training`.
- После выбора (или пропуска на hardcore) — §3.4.12 (владения), затем §3.4.11 (навыки) или сохранение.

#### 3.4.12. Владения (реализовано)
- Фиксированные владения из класса, расы и подкласса (если активен) начисляются автоматически.
- Выборы инструментов: класс (`proficiencies.tools.choice`), предыстория (`tool_proficiencies.choices`), подкласс (напр. battle master — ремесленные инструменты).
- Результат: `weapon_proficiencies`, `armor_proficiencies`, `tool_proficiencies` в JSON персонажа.
- UI: `ui/menus/proficiencies.py`; поздний выбор подкласса — `ui/menus/subclass_trainer.py`.
- См. [`05-equipment.md`](rules/05-equipment.md).

#### 3.4.11. Выбор навыков (реализовано)
- Объединение: расовые владения + навыки предыстории (из state) + выбор класса.
- При пересечении с классовым пулом — PHB swap (другой навык из пула класса).
- Компетентность (expertise) — отдельный шаг для классов с `expertise_count > 0` (напр. плут).
- После навыков (и expertise при необходимости) персонаж сохраняется в `saves/characters/{save_slug}.json`.

JSON персонажа включает `"languages": [...]`, `"background": "<id>"`, `"skills": [...]`, `"weapon_proficiencies"`, `"armor_proficiencies"`, `"tool_proficiencies"`.

### 3.5. Меню «Настройки» (реализовано)
- Заглушка: экран с пунктом «0. Назад».
- Выбор режима сложности — только в flow «Новая игра» / «Создать персонажа» (§3.3.1, §3.4.1).
- Язык интерфейса — в меню «Languages».

### 3.6. Меню «Languages» (реализовано)
- Самописная локализация на YAML-словарях (`load_strings`, `get_string`). Пакет `babel` в зависимостях не используется в коде.
- Язык по умолчанию: русский.
- Значки отображаются согласно текущей локализации:
  - При русской локализации:
    1. English
    2. Русский
    0. Назад
  - При английской локализации:
    1. Русский
    2. English
    0. Назад
- Тип ввода: **integer** по нумерованному списку.
- При смене языка интерфейс переключается с немедленным обновлением всех текстов.

### 3.7. Flow «Загрузить игру»
- Отображается список сохранённых приключений/сессий.
- Тип ввода: **integer** по нумерованному списку.
- Пункт "0" — Назад **всегда последний**.
- При выборе загружается состояние и запускается игровой цикл с момента сохранения.

## 4. Нефункциональные требования

### 4.1. Модульность
- Ядро (core) не зависит от UI, БД, модов.
- Core предоставляет API для:
  - Получения данных о персонаже
  - Выполнения действия (бросок кубика, проверка навыка, получение описания комнаты)
  - Управления состоянием игры (сохранение/загрузка)
- UI (меню, ввод/вывод) — отдельный слой.

### 4.2. Обработка ввода

**Основное правило:**  
По умолчанию от пользователя ожидается ввод **integer** соответствующего пункта нумерованного списка меню или списка действий. Первый пункт любого меню имеет номер "1", последний пункт "Выход" или "Назад" — номер "0". **Пункт "0" (Назад/Выход) должен отображаться всегда последним в любом меню.**

**Формат запроса:**  
При отрисовке меню используется строка вида:  
`Выберите пункт:`  

**Исключения (string ввод):**  
- Ввод имени персонажа при создании (не менее 2 символов, без пробелов и спецсимволов).
- Ввод пути к файлу мода (при необходимости).
- Ввод текста для поиска (в будущих итерациях).

**Валидация ввода:**  
- При integer вводе: проверка диапазона, повторный запрос при ошибке.
- При string вводе: проверка минимальной длины/формата, повторный запрос при ошибке.

### 4.3. Производительность
- Загрузка YAML-файлов при старте игры (или при смене языка/модов).
- Игровой цикл не должен задерживать ввод (отсутствие `time.sleep` без причины).

### 4.4. Локализация
- Все пользовательские тексты вынесены в YAML-словари (папка `database/strings/`).
- Ключи локализации используют точечную нотацию для вложенных словарей.
- При смене языка перезагружаются все текстовые ресурсы.
- Fallback на английский язык, если ключ не найден.

## 9. Ограничения и допущения

- Одиночная игра (без сети).
- Нет поддержки многопользовательской игры.
- Движок боя — упрощённый (D20 + модификаторы).
- Состояние игры сохраняется только вручную (автосохранение не требуется на Pre-Alpha).
- Графика отсутствует (только текст).
- Моды не требуют перезагрузки интерпретатора (но могут требовать перезагрузки ядра через вызов функции reload_mods).
